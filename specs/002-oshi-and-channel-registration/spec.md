## Clarifications

### Session 2026-06-20

- Q: 推しに最初のチャンネルを登録したとき、メインチャンネルの扱いは？ → A: 最初に登録したチャンネルを自動的にメインへ設定し、以後ユーザーが変更可能。チャンネルが1件以上ある限りメインは常に1つ存在する。
- Q: 推しの「テーマカラー」の入力形式は？ → A: 事前定義パレット（デザインで定めた固定色セット）からの選択とし、保存値はパレットの識別子に限定する（自由な HEX 入力は不可）。UI/デザイン面で色セットを事前定義する。
- Q: 通知可否トグルは本フィーチャーで配信まで実装するか？ → A: 設定値の保存のみ。実際の通知配信は後続フィーチャーが消費する前提とし、本フィーチャーでは配信を実装しない。

## User Scenarios & Testing *(mandatory)*

ログイン済みユーザーが、自分の「推し」を登録し、その推しの YouTube チャンネルを登録できるようにする。
これにより、後続フィーチャー（動画同期・新着一覧・視聴管理）が「誰の・どのチャンネルを追うのか」という
前提データを持てるようになる。本フィーチャーは推しとチャンネルの**登録・管理**までを担い、動画そのものの
取得・一覧表示は範囲外（後続フィーチャー）とする。

### User Story 1 - 推しを登録して管理する (Priority: P1)

VTuberファンであるユーザーが、自分の推しを名前（任意でグループ名・テーマカラー・メモ）とともに登録し、
自分の推し一覧として管理できる。

**Why this priority**: 「推し」はチャンネル登録・動画整理の起点となる単位。これが無いと以降のチャンネル紐付け
が成立しないため最優先。推しを1人登録できるだけでも「自分の推しを記録する」という最小の価値を提供する。

**Independent Test**: ログイン後に推しを1人作成し、自分の推し一覧に表示され、編集・削除できることを確認すれば
単独でテスト・デモ可能。

**Acceptance Scenarios**:

1. **Given** ログイン済みユーザー, **When** 名前を入力して推しを作成する, **Then** その推しが自分の推し一覧に表示される
2. **Given** 推しを登録済みのユーザー, **When** グループ名・テーマカラー・メモを編集する, **Then** 変更が保存され一覧/詳細に反映される
3. **Given** 推しを登録済みのユーザー, **When** 推しを削除する, **Then** その推しと、それに紐づく自分のチャンネル登録が一覧から消える
4. **Given** 名前未入力, **When** 推しの作成を試みる, **Then** 作成は拒否され、入力を促す案内が表示される
5. **Given** 別ユーザーが登録した推し, **When** その推しを参照・編集しようとする, **Then** アクセスは拒否される

---

### User Story 2 - 推しの YouTube チャンネルを登録する (Priority: P1)

ユーザーが、推しの YouTube チャンネルを URL または @handle で入力して登録できる。サービスは入力から
チャンネルを一意に特定し、チャンネル情報を取得して、ユーザーの登録として保存する。

**Why this priority**: チャンネルの登録は、後続の「新着動画の取得・整理」の必須入力。推し登録と並ぶ本フィーチャー
の中核価値であり、これが成立して初めてサービスの目的（推しの配信を追う）に近づく。

**Independent Test**: 有効なチャンネル URL または @handle を入力して登録し、対象チャンネルが自分の登録一覧に
表示されること、無効な入力が拒否されることを確認すれば単独でテスト可能（動画取得自体は範囲外）。

**Acceptance Scenarios**:

1. **Given** ログイン済みユーザーと登録済みの推し, **When** チャンネル URL（`/channel/UC...` または `/@handle`）を入力して登録する, **Then** チャンネルが特定され、推しに紐づくチャンネル登録として保存される
2. **Given** @handle 形式（`@handle` または handle の URL）の入力, **When** 登録する, **Then** チャンネルが特定され登録される
3. **Given** 対応形式以外の文字列, **When** 登録を試みる, **Then** 登録は拒否され、対応形式を案内するメッセージが表示される
4. **Given** 実在しない／特定できないチャンネルの入力, **When** 登録を試みる, **Then** 登録は拒否され、分かりやすい案内が表示される
5. **Given** すでに自分が登録済みのチャンネル, **When** 同じチャンネルを再登録しようとする, **Then** 重複登録は防止され、その旨が案内される
6. **Given** 他ユーザーがすでに登録済みのチャンネル, **When** 自分が同じチャンネルを登録する, **Then** チャンネル情報は共有され重複作成されず、自分の登録だけが新たに作られる
7. **Given** YouTube 側が一時的に応答しない／クォータ超過, **When** 登録を試みる, **Then** サービスは停止せず、再試行を促すユーザー向けメッセージが表示される

---

### User Story 3 - 登録した推しとチャンネルを一覧・確認する (Priority: P2)

ユーザーが、自分が登録した推しと、それぞれに紐づく YouTube チャンネルを一覧・詳細で確認できる。

**Why this priority**: 登録結果を確認できることは管理の前提だが、作成（US1/US2）が先に成立している必要がある
ためP2。後続の動画一覧の土台にもなる。

**Independent Test**: 推しとチャンネルを登録した状態で、推し一覧・推し詳細にチャンネルが表示され、チャンネルの
同期状態（同期待ち等）が確認できることを検証できる。

**Acceptance Scenarios**:

1. **Given** 推しとチャンネルを登録済みのユーザー, **When** 推し一覧を開く, **Then** 各推しと紐づくチャンネル（数や代表チャンネル）が表示される
2. **Given** 推し詳細を開く, **When** 表示する, **Then** その推しに紐づくチャンネルの一覧（タイトル・サムネ・同期状態）が表示される
3. **Given** 登録直後のチャンネル, **When** 詳細を確認する, **Then** 「同期待ち（動画はこれから取得）」の状態が分かる
4. **Given** 別ユーザーの推し詳細 URL, **When** 直接アクセスする, **Then** 参照は拒否される

---

### User Story 4 - チャンネルの同期・通知設定とメインチャンネルを管理する (Priority: P2)

ユーザーが、登録済みチャンネルごとに「同期対象にするか」「通知を受けるか」を切り替え、推しの
メインチャンネルを指定できる。また不要になったチャンネル登録を解除できる。

**Why this priority**: 同期・通知の制御は運用上必要だが、登録（US1/US2）が前提のためP2。後続の同期処理が
「どのチャンネルを対象にするか」を決める設定値を本フィーチャーで確立する。

**Independent Test**: 登録済みチャンネルの同期/通知トグルを切り替え、メインチャンネルを指定し、登録解除が
できることを確認できる。

**Acceptance Scenarios**:

1. **Given** 登録済みチャンネル, **When** 同期を無効にする, **Then** 設定が保存され、そのチャンネルは以後の同期対象から外れる扱いになる
2. **Given** 推しに複数チャンネルを登録済み, **When** 1つをメインチャンネルに指定する, **Then** その推しのメインチャンネルは常に1つだけになる
3. **Given** 登録済みチャンネル, **When** 通知設定を切り替える, **Then** 設定が保存される
4. **Given** 不要なチャンネル登録, **When** 登録を解除する, **Then** 自分の登録のみが削除され、共有のチャンネル情報や他ユーザーの登録は影響を受けない

---

### Edge Cases

- 対応形式以外（YouTube 以外の URL、空文字、動画 URL 等）が入力された場合は、登録せず対応形式を案内する。
- 入力からチャンネルを一意に特定できない（存在しない・削除済み・非公開）場合は、登録せず分かりやすく案内する。
- 同一ユーザーが同一チャンネルを重複登録しようとした場合は防止する（一意制約）。
- 同一チャンネルを複数ユーザーが登録する場合、チャンネル情報は全ユーザー共通の1レコードを共有し、複製しない。
- MVP では、同一ユーザーが同一チャンネルを複数の推しに同時に紐付けることはできない。
- 1人の推しに複数チャンネルを紐付けられる。最初に登録したチャンネルが自動的にメインになり、推しにチャンネルが1件以上ある限りメインは常にちょうど1つ存在する（ユーザーが変更可能）。
- 推しを削除した場合、その推しに紐づく「自分のチャンネル登録」も削除する（共有のチャンネル情報は残す）。
- YouTube API がエラー／クォータ超過の場合でも、サービスはエラーで停止せず再試行を促す。
- handle の表記揺れ（`@` の有無・大文字小文字・末尾スラッシュ）は正規化して特定を試みる。
- 登録は認証必須であり、未ログイン時はログインへ誘導される。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、ユーザーが推しを作成できる手段を提供しなければならない（名前は必須、グループ名・テーマカラー・メモは任意）。テーマカラーは、デザインで事前定義した固定パレットからの選択とし、保存値はパレットの識別子に限定する（自由な HEX 入力は受け付けない）。パレット外の値は拒否する。
- **FR-002**: ユーザーは、自分の推しを一覧・詳細で参照し、編集・削除できなければならない。
- **FR-003**: システムは、ユーザーが YouTube チャンネルを URL または @handle で入力して登録できる手段を提供しなければならない。対応形式は少なくとも「チャンネル URL（`/channel/UC...`）」「@handle の URL」「@handle 文字列」とする。
- **FR-004**: システムは、入力からチャンネルを一意に特定（解決）しなければならない。対応形式外の入力、および特定できない入力は登録せず、ユーザー向けの分かりやすい案内を返さなければならない。
- **FR-005**: システムは、特定したチャンネルの公開メタ情報（タイトル・説明・ハンドル・サムネイル・投稿動画一覧の参照情報・公開日時等）を取得し、全ユーザー共通の共有マスタとして重複なく保持しなければならない。
- **FR-006**: システムは、ユーザーが推しに対してチャンネルを登録できるようにし、ユーザー固有の登録情報（対象の推し・同期対象可否・通知可否・メイン指定・登録日時）を保持しなければならない。
- **FR-007**: システムは、同一ユーザーが同一チャンネルを重複登録できないようにしなければならない。
- **FR-008**: システムは、1人の推しに複数のチャンネルを紐付けられるようにしなければならない。ただし MVP では、同一ユーザーが同一チャンネルを複数の推しに同時に紐付けることを許可してはならない。
- **FR-009**: システムは、推しごとにメインチャンネルを管理しなければならない。推しに最初のチャンネルを登録したときは、そのチャンネルを自動的にメインに設定する。以後ユーザーはメインチャンネルを変更でき、推しにチャンネルが1件以上ある限り、メインチャンネルは常にちょうど1つ存在する。
- **FR-010**: システムは、ユーザーが自分の推し・チャンネル登録のみを参照・操作でき、他ユーザーのものへアクセスできないようにしなければならない。
- **FR-011**: システムは、共有マスタであるチャンネル情報を一般ユーザーから直接編集・削除できないようにしなければならない（更新は同期・管理処理に限る）。
- **FR-012**: ユーザーは、登録済みチャンネルごとに同期対象可否・通知可否を切り替えられなければならない。通知可否は本フィーチャーでは設定値の保存のみを行い、実際の通知配信は後続フィーチャーが本設定値を消費して実装する（本フィーチャーでは配信を行わない）。
- **FR-013**: システムは、登録直後のチャンネルを「同期待ち」の状態として扱い、動画そのものの取得・一覧表示は後続フィーチャー（動画同期）に引き継がなければならない。
- **FR-014**: システムは、YouTube 側のエラー・クォータ超過時でも停止せず、内部情報を露出しないユーザー向けメッセージを返さなければならない。
- **FR-015**: システムは、推し作成・チャンネル登録・一覧確認の主要操作を、PC・スマートフォンのブラウザで完了できるようにしなければならない（レスポンシブ対応）。
- **FR-016**: システムは、推し操作・チャンネル登録など認証済み操作に対して濫用防止のためのレート制限を適用しなければならない。
- **FR-017**: ユーザーは、不要になったチャンネル登録を解除でき、その際に削除されるのは自分の登録のみで、共有のチャンネル情報や他ユーザーの登録は影響を受けてはならない。

### Key Entities *(include if feature involves data)*

- **推し（Oshi）**: ユーザーが追う対象の単位。所有者であるユーザーに紐づく。名前（必須）、グループ名・テーマカラー・メモ（任意）を持つ。テーマカラーは事前定義パレットの識別子として保持する（自由な色文字列ではない）。1人の推しは複数のチャンネル登録を持ちうる。
- **YouTubeチャンネル（共有マスタ）**: 全ユーザー共通のチャンネル情報。YouTube 上のチャンネル識別子で一意。タイトル・説明・ハンドル・サムネイル・投稿動画一覧の参照情報・公開日時・同期状態などを持つ。ユーザー単位で複製しない。
- **チャンネル登録（UserChannel）**: 「どのユーザーが・どの推しに・どのチャンネルを」登録したかという関連と、ユーザー固有の設定（同期対象可否・通知可否・メイン指定・登録日時）を保持する。ユーザー所有データであり、所有権で保護される。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが、推しの作成からチャンネル登録完了までを、通常2分以内・数ステップで完了できる。
- **SC-002**: 対応形式のチャンネル URL／@handle の95%以上について、正しいチャンネルを特定して登録できる。
- **SC-003**: 同一ユーザーによる同一チャンネルの重複登録が成立する割合は0件である。
- **SC-004**: 同一チャンネルを複数ユーザーが登録しても、保持されるチャンネル情報のレコードは常に1件（同一）である。
- **SC-005**: あるユーザーが別ユーザーの推し・チャンネル登録に到達できる割合は0件である。
- **SC-006**: YouTube 側の障害・クォータ超過時でも、画面がエラーで停止せず、再試行を促す案内が表示される。
- **SC-007**: PC・スマートフォンの双方で、推し作成・チャンネル登録・一覧確認の主要操作を完了できる。
- **SC-008**: 推しにチャンネルが1件以上登録されている場合、メインチャンネルは常にちょうど1つ存在する（複数登録時も指定は常に1つ）。

## Assumptions

- 本フィーチャーは Feature 1（認証基盤・Supabase Auth・profiles・所有権 Policy 土台）が完成していることを前提とする。ユーザー識別子は profiles の UUID を用いる。
- 動画そのものの取得（初回同期・最新動画の収集）、新着一覧、視聴ステータス管理は本フィーチャーの範囲外であり、後続フィーチャー（003 video-sync 以降）で扱う。本フィーチャーは登録までを担い、チャンネルを「同期待ち」状態にする。初回同期処理の起動・実行は後続フィーチャーで実装する。
- 通知（メール／プッシュ等）の配信も本フィーチャーの範囲外とする。本フィーチャーは通知可否の設定値を保持するのみで、配信ロジックは後続フィーチャーが本設定値を消費して実装する。
- チャンネルの特定・情報取得は、憲法V（クォータ規律）に従い `search.list` を用いず、最小限の API 呼び出しで行う。投稿動画の収集には uploads playlist の参照情報を用いる（取得自体は後続）。
- MVP では、チャンネルを登録する際に対象の推しを選択（または同時に作成）して紐付ける。「推し未分類」でチャンネルだけ登録する運用は将来対応とする。
- 推しを削除すると、その推しに紐づく自分のチャンネル登録も削除する。共有マスタのチャンネル情報は他ユーザーの登録のために残す。
- 取得・保持するのはチャンネルの公開メタ情報に限り、動画ファイル・サムネイル画像本体・API レスポンス全文は永続保存しない（憲法II）。
- 公開前に必要な利用規約・プライバシーポリシー等の整備は最終フィーチャー（hardening）で扱う。
