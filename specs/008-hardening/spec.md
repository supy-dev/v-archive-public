**Feature Branch**: `008-hardening`

**Created**: 2026-06-22

**Status**: Draft

## 情報設計の前提

本 Feature は機能追加ではなく「公開品質への引き上げ」を目的とする。Phase 1〜6（Feature 1〜7）で実装済みの全機能を対象に、セキュリティ・堅牢性・パフォーマンス・法的整備を横断的に強化する。

| 区分 | 内容 |
|------|------|
| **セキュリティ** | 認可 Policy の全カバレッジ確認・レート制限・秘密情報管理 |
| **エラー処理** | ユーザー向けエラー画面・内部ログ規律・API 障害フォールバック |
| **パフォーマンス** | N+1 解消・インデックス確認・ページネーション |
| **テスト** | E2E テスト追加・既存 Feature/Unit テストの網羅性確認 |
| **規約整備** | プライバシーポリシー・利用規約ページの公開 |

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 不正なデータ操作から自分のデータを守る (Priority: P1)

ログイン中のユーザーが他ユーザーのメモ・ウォッチアイテム・チャンネルを参照・編集・削除できないことを保証する。すべての書き込み操作で所有権確認が行われ、不正アクセスは 403 で拒否される。

**Why this priority**: 所有権チェック漏れは公開後の重大インシデントに直結する。全機能実装が揃ったこのタイミングで横断確認し、穴を塞ぐ必要がある。

**Independent Test**: 認証済みの別ユーザーとして、他ユーザーのメモ ID に対して更新・削除リクエストを送ると 403 を受け取る。

**Acceptance Scenarios**:

1. **Given** ユーザーAが自分のタイムスタンプメモを持つとき、**When** ユーザーBがそのメモを更新しようとすると、**Then** 403 Forbidden が返り、メモは変更されない。
2. **Given** ユーザーAのウォッチアイテムが存在するとき、**When** ユーザーBがそのアイテムのステータスを変更しようとすると、**Then** 403 Forbidden が返る。
3. **Given** 未認証ユーザーがアプリ内の保護ページへアクセスするとき、**When** URL を直接入力すると、**Then** ログイン画面へリダイレクトされる。
4. **Given** 認証済みユーザーが共有マスタ（youtube_channels）を直接更新しようとするとき、**When** リクエストを送ると、**Then** 403 または 404 が返り、更新は行われない。

---

### User Story 2 - 過剰なリクエストからサービスを守る (Priority: P1)

手動チャンネル同期・メモ登録・ステータス変更などの書き込み操作に対して、短時間での連打やスクリプト攻撃を防ぐレート制限が適用されている。

**Why this priority**: 手動同期は YouTube API クォータを消費する。制限なしでは単一ユーザーがクォータを枯渇させてサービス全体に影響する。

**Independent Test**: 手動チャンネル同期を短時間に規定回数以上送信すると、「しばらくしてからお試しください」というメッセージと共に操作が拒否される。

**Acceptance Scenarios**:

1. **Given** ユーザーが手動チャンネル同期を短時間に繰り返すとき、**When** 規定回数（例: 1分あたり5回）を超えると、**Then** 「しばらくしてからお試しください」と表示され操作が止まる。
2. **Given** 認証済みルートへの書き込みリクエストが規定数を超えたとき、**When** 超過リクエストが来ると、**Then** 429 Too Many Requests を返す。
3. **Given** レート制限が発動したとき、**When** 待機時間が経過すると、**Then** 通常通り操作できる。

---

### User Story 3 - エラー発生時も安心して使い続けられる (Priority: P2)

ページが見つからない（404）・サーバーエラー（500）・YouTube API 障害などが発生しても、ユーザーはアプリデザインに沿ったエラー画面を表示され、保存済みデータへのアクセスは維持される。

**Why this priority**: 外部 API 障害は避けられない。API が落ちてもメモ・一覧閲覧が使えることは仕様書（§21）の完了条件に明記されている。

**Independent Test**: YouTube API の応答をオフ（モック）にした状態でアプリを操作し、保存済み動画一覧・タイムスタンプメモの閲覧が正常に機能することを確認する。

**Acceptance Scenarios**:

1. **Given** 存在しない URL を開くとき、**When** ページを表示すると、**Then** アプリのデザインに沿った 404 ページが表示され、ホームへ戻るリンクがある。
2. **Given** サーバー側で予期しないエラーが発生したとき、**When** 該当画面を開くと、**Then** アプリのデザインに沿った 500 ページが表示され、スタックトレースがユーザーには見えない。
3. **Given** YouTube API が利用不可のとき、**When** 保存済み動画一覧を開くと、**Then** 「YouTubeとの同期に一時的に失敗しました。保存済みの動画一覧は引き続き利用できます。」と表示され、一覧自体は閲覧できる。
4. **Given** チャンネル同期が YouTube API エラーで失敗したとき、**When** チャンネル詳細を確認すると、**Then** エラー内容の技術情報（APIキー等）はユーザーに表示されない。

---

### User Story 4 - 大量データでもスムーズに閲覧できる (Priority: P2)

神回動画・タイムスタンプメモ・新着アーカイブが数百件〜数千件に増えても、一覧画面が遅くなったり、データベースへ過大な負荷をかけたりしない。

**Why this priority**: MVP 公開後にデータが蓄積してから対処すると既存ユーザーへの影響が大きい。事前に N+1 や欠落インデックスを解消しておく。

**Independent Test**: 動画 500 件・メモ 1000 件のシードデータを投入した状態で新着一覧・メモ一覧を開き、3秒以内に表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 動画一覧に 500 件以上のデータがあるとき、**When** 新着一覧ページを開くと、**Then** 3 秒以内に最初のページが表示される。
2. **Given** タイムスタンプメモが 1000 件以上あるとき、**When** `/memos` を開くと、**Then** 20 件単位でページネーションされ、初期表示は 3 秒以内に完了する。
3. **Given** 一覧ページを表示するとき、**When** 実行される DB クエリを確認すると、**Then** N+1 問題のない件数のクエリが発行されている（必要なリレーションが一括ロードされている）。

---

### User Story 5 - サービス利用規約とプライバシーポリシーを確認できる (Priority: P3)

ユーザーが個人情報の取り扱いやサービスルールを確認したいとき、アプリ内でプライバシーポリシーと利用規約が閲覧できる。

**Why this priority**: 公開前に法的ページを用意することは §17 の完了条件。P1/P2 の技術強化後に対応する。

**Independent Test**: 未ログイン状態でフッターのリンクをクリックし、プライバシーポリシーページと利用規約ページが表示される。

**Acceptance Scenarios**:

1. **Given** アプリのフッターを確認するとき、**When** 「プライバシーポリシー」リンクをクリックすると、**Then** プライバシーポリシーページが表示される（ログイン不要）。
2. **Given** アプリのフッターを確認するとき、**When** 「利用規約」リンクをクリックすると、**Then** 利用規約ページが表示される（ログイン不要）。
3. **Given** 両ページを確認するとき、**When** ページを開くと、**Then** 収集する情報・利用目的・免責事項が記載されており、最終更新日が明記されている。

### Edge Cases

- YouTube API が不安定（断続的にエラー）な場合でも、同期 Job のリトライが過剰に API クォータを消費しない。
- レート制限はユーザー単位で適用され、他ユーザーの操作に影響しない。
- 500 エラーページが自身のレンダリングでエラーを起こさない（ネストしたエラーを防ぐ）。
- インデックス追加によって既存データへの書き込み速度が大きく低下しない。
- 規約ページは認証状態に関わらずアクセスできる（プライベートルート外）。
- テストでレート制限が誤発動して CI が落ちない（テスト環境では制限を無効化またはリセット可能にする）。
- E2E テストが実際の YouTube API を呼ばず、完全にモック・スタブで完結する。

## Requirements *(mandatory)*

### Functional Requirements

**認可・セキュリティ（Policy）**

- **FR-001**: 全ユーザー所有データ（oshis・user_channels・user_watch_items・video_notes・timestamp_memos・ユーザー固有タグ）への書き込み操作は、Laravel Policy による所有権確認を通過した場合のみ許可されなければならない（MUST）。
- **FR-002**: 共有マスタ（youtube_channels・youtube_videos）への一般ユーザーからの直接更新を、Policy または Route 設計によって禁止しなければならない（MUST NOT）。
- **FR-003**: 読み取り操作（一覧・詳細閲覧）にはレート制限を設けず、書き込み操作（メモ作成・更新・削除、ステータス変更、神回トグル等）に対してユーザー単位で 1 分あたり 30 回の制限を設け、超過時は 429 で拒否しなければならない（MUST）。
- **FR-004**: 手動チャンネル同期エンドポイントには書き込み操作共通制限（30回/分）よりさらに厳しい個別制限（1分あたり5回）を適用しなければならない（MUST）。
- **FR-005**: ログ出力において、Supabase アクセストークン・YouTube APIキー・service role key・認証Cookie・メモ本文の不要な全文を記録してはならない（MUST NOT）。

**エラー処理**

- **FR-006**: アプリケーションは、アプリのデザインに沿った 404 エラーページを提供しなければならない（MUST）。
- **FR-007**: アプリケーションは、スタックトレースを含まない 500 エラーページを提供しなければならない（MUST）。
- **FR-008**: YouTube API が応答しない場合でも、保存済み動画一覧・タイムスタンプメモ・神回一覧の閲覧を維持しなければならない（MUST）。
- **FR-009**: API 障害時のユーザー向けメッセージは「YouTubeとの同期に一時的に失敗しました。保存済みの動画一覧は引き続き利用できます。」の形式に従わなければならない（MUST）。
- **FR-010**: 内部ログには、チャンネルID・動画ID・Job名・HTTPステータス・APIエラー理由・リトライ回数・処理時間を記録しなければならない（MUST）。

**パフォーマンス**

- **FR-011**: 全一覧画面でページネーション（20件/ページ）を使用し、全件ロードを行ってはならない（MUST）。
- **FR-012**: 一覧画面のクエリは N+1 問題を解消し、必要なリレーションを一括ロードしなければならない（MUST）。
- **FR-013**: 推奨インデックス（`youtube_videos(youtube_channel_id, published_at)`・`user_watch_items(user_id, status)`・`user_watch_items(user_id, updated_at)`・`timestamp_memos(user_id, youtube_video_id, seconds)`・`user_channels(user_id, is_sync_enabled)`）が存在しなければならない（MUST）。

**テスト**

- **FR-014**: 他ユーザーのデータへの不正アクセスを 403 で拒否することを確認する Feature テストが存在しなければならない（MUST）。
- **FR-015**: レート制限が発動した場合に 429 を返すことを確認するテストが存在しなければならない（MUST）。
- **FR-016**: YouTube API をモックした状態で API 障害フォールバックを確認するテストが存在しなければならない（MUST）。
- **FR-017**: E2E テストは YouTube 実 API を呼ばず、完全にモック・スタブで実行されなければならない（MUST）。

**規約ページ**

- **FR-018**: プライバシーポリシーページ（`/privacy`）が、ログイン不要で閲覧できなければならない（MUST）。
- **FR-019**: 利用規約ページ（`/terms`）が、ログイン不要で閲覧できなければならない（MUST）。
- **FR-020**: 両ページには、収集する情報・利用目的・免責事項・最終更新日が記載されなければならない（MUST）。
- **FR-021**: アプリのフッターにプライバシーポリシーと利用規約へのリンクを設けなければならない（MUST）。

### Key Entities

- **ユーザー所有データ群**: oshis・user_channels・user_watch_items・video_notes・timestamp_memos・ユーザー固有タグ。Policy が所有権を保護する対象。
- **共有マスタ**: youtube_channels・youtube_videos。一般ユーザーからの直接書き込みを拒否する対象。
- **エラーログ**: チャンネルID・動画ID・Job名・HTTPステータス・エラー理由・リトライ回数・処理時間を記録する。秘密情報を含めない。
- **規約ページ**: プライバシーポリシーと利用規約の静的コンテンツ。認証不要でアクセス可能。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 別ユーザーのデータに対する不正な書き込みリクエストが、100% の割合で 403 Forbidden で拒否される。
- **SC-002**: 書き込み操作（メモ作成・ステータス変更・神回トグル等）が 1 分あたり 30 回を超えた場合に 429 で拒否される。手動チャンネル同期は 1 分あたり 5 回を超えた場合に 429 で拒否される。
- **SC-003**: YouTube API が完全に停止している状態でも、保存済み動画一覧・タイムスタンプメモ一覧・神回一覧が 3 秒以内に表示される。
- **SC-004**: 動画 500 件・メモ 1000 件のデータ量で、一覧初期表示が 3 秒以内に完了する。
- **SC-005**: 一覧クエリで発行される SQL クエリ数が、表示件数に比例して増加しない（N+1 の解消）。
- **SC-006**: 404・500 エラー画面がアプリのデザインに沿って表示され、スタックトレースを含まない。
- **SC-007**: プライバシーポリシー・利用規約ページが未認証状態で閲覧でき、各ページに最終更新日が記載されている。
- **SC-008**: 全 Feature テスト・Unit テスト・E2E テストが CI 上で YouTube 実 API なしに通過する。

## Clarifications

### Session 2026-06-22

- Q: 手動同期のレート制限は何回/何分とするか？ → A: 1 ユーザーあたり 1 分あたり 5 回を初期値とし、調整可能な設定値として管理する。
- Q: E2E テストの対象範囲はどこまでか？ → A: ログイン → チャンネル登録 → 新着一覧 → 動画再生 → メモ保存 の主要フローを最低限カバーする。ブラウザ操作の自動化（Playwright/Dusk等）は実装手段として検討するが、仕様には含めない。
- Q: プライバシーポリシー・利用規約の文面は誰が作成するか？ → A: 本仕様ではページの存在と最低掲載事項（収集情報・目的・免責・更新日）のみを定義する。文面の法的レビューは本機能スコープ外とし、「ドラフト版」として公開して後日更新する前提とする。
- Q: 認証済みルート全体のレート制限（FR-003）はどの単位・制限値を基本とするか？ → A: 読み取り操作は制限なし、書き込み操作のみ 1 分あたり 30 回のユーザー単位制限を設ける。手動同期は個別に 1 分あたり 5 回の厳しい制限を適用する。

## Assumptions

- 全 Policy クラス（UserWatchItemPolicy・TimestampMemoPolicy 等）は Feature 1〜6 で作成済みであり、本 Feature では所有権チェックの漏れ確認と未実装箇所の補完を行う。
- レート制限は Laravel の組み込み機能（RateLimiter ファサード）で実装し、外部サービスを追加しない。
- エラーページ（404・500）はアプリのデザインガイドライン（CLAUDE.md）に沿ったカスタムビューとして実装する。
- パフォーマンステストはシードデータを使った Feature テストで代替し、本番相当の負荷試験ツールは本機能スコープ外とする。
- 推奨インデックスはマイグレーションとして追加し、本番データへの影響を考慮した `CONCURRENTLY`（または同等の安全な方法）を用いることが望ましいが、実装詳細は plan フェーズで判断する。
- プライバシーポリシー・利用規約ページは静的コンテンツとして実装し、CMS や管理画面からの編集機能は本機能スコープ外とする。
- フッターへのリンク追加は `layouts/app.blade.php` の変更で対応し、既存レイアウトのデザインを維持する。
- テストでのレート制限は、テスト用 `WithoutMiddleware` または `RateLimiter::clear()` を使用して無効化・リセット可能にする。
- 本 Feature は新規の DB テーブルやカラムを追加しない（インデックスのみ追加する）。
