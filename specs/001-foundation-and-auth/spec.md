**Status**: Draft

## User Scenarios & Testing *(mandatory)*

VTuberアーカイブ手帳の基盤となる、ユーザー認証とアカウント基盤を提供する。本サービスのすべての
機能はログインしたユーザーに紐づくため、本フィーチャーは以降のすべてのフィーチャーの前提となる。

### User Story 1 - 推しの管理を始めるためにアカウント登録・ログインする (Priority: P1)

VTuberファンであるユーザーが、サービスのトップにアクセスし、Googleアカウント、または
メールアドレスとパスワードでアカウントを作成・ログインして、自分専用のホーム画面に入れる。
これにより、以降の推し登録・動画整理を自分のデータとして始められる。

**Why this priority**: ログインできなければ本サービスのいかなる機能も利用できない。これがMVP全体の
入り口であり、最初に動作確認できる最小の価値（＝自分のスペースに入れること）を提供する。

**Independent Test**: 未登録ユーザーがGoogleまたはメールアドレスとパスワードでアカウントを作成し、
ログイン後に認証済みホーム画面が表示されることを確認するだけで、単独でテスト・デモ可能。

**Acceptance Scenarios**:

1. **Given** 未ログインの訪問者がトップにアクセスした, **When** 「ログイン」を選びGoogle認証を完了する, **Then** 認証済みホーム画面が表示され、本人として識別される
2. **Given** アカウント未作成の訪問者, **When** メールアドレスとパスワード（8文字以上）で新規登録する, **Then** アカウントが作成され、登録メールアドレス宛に確認メールが送られ、確認完了が促される
3. **Given** メール確認が未完了のユーザー, **When** ログインまたは保護画面の利用を試みる, **Then** 利用は許可されず、メール確認を促す案内（確認メール再送を含む）が表示される
4. **Given** 確認メールのリンクをクリックして確認を完了したユーザー, **When** メールアドレスと正しいパスワードでログインする, **Then** 前回と同一のアカウント（同じプロフィール）として認識され、ホーム画面に入れる
5. **Given** メール確認済みで登録済みのユーザー, **When** 誤ったパスワードでログインを試みる, **Then** ログインは成立せず、どの項目が不正かを過度に開示しない案内が表示される
6. **Given** 過去にログインしたことのあるユーザー, **When** 再度ログインする, **Then** 前回と同一のアカウント（同じプロフィール）として認識される
7. **Given** 認証プロバイダ側で認証をキャンセル／失敗した, **When** サービスに戻る, **Then** ログインは成立せず、分かりやすい案内とともに未ログイン状態に留まる

---

### User Story 4 - パスワードを忘れても再びログインできる (Priority: P2)

メール＋パスワードで登録したユーザーが、パスワードを忘れた場合に、登録メールアドレス宛の
再設定手続きを通じてパスワードを再設定し、再びログインできる。

**Why this priority**: パスワード方式を採用する以上、再設定手段がないとアカウントへ復帰できず
利用継続が破綻する。ただしログイン・登録そのものより後でよいためP2。

**Independent Test**: 登録済みユーザーが「パスワードをお忘れですか」から再設定を実行し、新しい
パスワードでログインできることを確認できる。

**Acceptance Scenarios**:

1. **Given** メール＋パスワードで登録済みのユーザー, **When** 登録メールアドレスでパスワード再設定を申請する, **Then** 当該メール宛に再設定手段が送られる
2. **Given** 有効な再設定手段を受け取ったユーザー, **When** 新しいパスワードを設定する, **Then** 新しいパスワードでログインでき、古いパスワードは無効になる
3. **Given** 未登録または存在しないメールアドレス, **When** 再設定を申請する, **Then** アカウントの有無を推測させない一様な案内が表示される

---

### User Story 2 - 自分のデータを他人から守る (Priority: P1)

ログインしていない、または別人のユーザーが、保護された画面や他人のデータにアクセスできない。
ユーザーは自分のデータだけを閲覧・操作できる。

**Why this priority**: マルチユーザーサービスとして、認証境界と所有権の保証は最初から成立して
いなければならない（憲法 III）。後付けで漏れると全ユーザーに影響する。

**Independent Test**: 未ログイン状態で保護URLにアクセスするとログインへ誘導されること、別ユーザーの
識別子では自分のデータにアクセスできないことを確認できる。

**Acceptance Scenarios**:

1. **Given** 未ログインの訪問者, **When** 認証が必要な画面のURLへ直接アクセスする, **Then** ログイン画面へ誘導され、保護コンテンツは表示されない
2. **Given** 有効期限切れ・改ざんされた認証情報を持つリクエスト, **When** 保護画面へアクセスする, **Then** 認証は拒否され、未ログイン扱いになる
3. **Given** ログイン済みユーザーA, **When** ユーザーBのものとして識別を偽装しようとする, **Then** サービスはサーバー側で確立した本人の識別のみを採用し、偽装は無効になる

---

### User Story 3 - 自分の表示情報を確認しログアウトする (Priority: P2)

初回ログイン時に、ユーザーの表示名・アイコン・タイムゾーンを含むプロフィールが自動的に用意され、
ユーザーはそれを確認できる。利用を終えるときは確実にログアウトできる。

**Why this priority**: 推し別の振り返りや日時表示の前提となるプロフィール（特にタイムゾーン）を
最初に確立する。ログアウトは共有端末での安全な利用に必要だが、ログイン自体より後でよい。

**Independent Test**: 初回ログイン後にプロフィールが存在し表示名等が初期化されていること、
ログアウト後に保護画面へアクセスできなくなることを確認できる。

**Acceptance Scenarios**:

1. **Given** 初めてログインしたユーザー, **When** ログインが完了する, **Then** 表示名・アイコン・既定タイムゾーンを持つプロフィールが自動的に作成される
2. **Given** ログイン済みユーザー, **When** ログアウトを実行する, **Then** セッションが破棄され、以降の保護画面アクセスはログインを要求される
3. **Given** ログアウト済みのユーザー, **When** ブラウザの戻る操作で保護画面を再表示しようとする, **Then** 保護コンテンツは表示されない

---

### Edge Cases

- 認証プロバイダ（Supabase Auth）が一時的に利用できない場合、ユーザーには再試行を促す案内を表示し、サービスはエラーで停止しない。
- 既にメール＋パスワードで登録済みのメールアドレスで再登録を試みた場合は、アカウントの有無を不正に開示せず重複作成を防ぐ。
- 既存のGoogleアカウントと同一メールアドレスでメール＋パスワード登録（またはその逆）が行われた場合は、同一ユーザー（同一プロフィール）にリンクし、双方の方式から同じアカウントへログインできるようにする。
- メール確認が未完了のユーザーがログインを試みた場合は、確認を促し確認メールの再送手段を提供する。確認リンクの有効期限が切れている場合は再送を促す。
- パスワードログインの連続失敗に対しては、濫用防止のためのレート制限・抑制を適用する。
- パスワード再設定手段の有効期限が切れている場合、再申請を促す。
- 認証情報の有効期限が利用中に切れた場合、次の保護操作時に再ログインへ誘導する。
- 同一ユーザーが複数端末／タブで同時にログインしている場合でも、同一プロフィールとして一貫して扱う。
- 認証は成立したがプロフィール作成に失敗した場合でも、次回アクセス時に整合が取れるよう冪等に補完する。
- ログアウト時は、サービス側セッションと認証プロバイダ側セッションの双方が破棄される。

## Clarifications

### Session 2026-06-19

- Q: メール＋パスワード新規登録時、メール確認をログイン前に必須にするか → A: 必須（確認完了後に利用可）
- Q: 同一メールアドレスがGoogleとメール＋パスワード双方で使われた場合の扱い → A: 同一アカウントにリンク（同一プロフィールを共有）
- Q: パスワードの最小文字数（強度要件） → A: 8文字以上

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、ユーザーが外部認証プロバイダ（Supabase Auth）経由で、(a) Googleアカウント、および (b) メールアドレスとパスワード、のいずれの方法でもログインできる手段を提供しなければならない。
- **FR-001a**: システムは、ユーザーがメールアドレスとパスワードで新規アカウントを登録できる手段を提供しなければならない。同一メールアドレスでの重複登録は防止すること。
- **FR-001b**: システムは、メール＋パスワードのユーザーが、登録メールアドレス宛の手続きを通じてパスワードを再設定できる手段を提供しなければならない。パスワード再設定の申請に対しては、アカウントの有無を推測させない一様な応答を返すこと。
- **FR-001c**: システムは、パスワードを8文字以上とする強度要件を課し、平文で保存・ログ出力してはならない（保管・ハッシュ化は認証プロバイダに委ねる）。8文字未満のパスワードは登録・再設定時に拒否すること。
- **FR-001d**: システムは、メールアドレス＋パスワードで新規登録したユーザーに対してメール確認を必須とし、登録メールアドレスの確認が完了するまでログインおよび保護機能の利用を許可してはならない。確認メールの再送手段を提供すること。
- **FR-002**: システムは、認証が必要なすべての画面・操作について、有効な認証が確立されている場合のみアクセスを許可しなければならない。
- **FR-003**: システムは、提示された認証情報の正当性（発行元・有効期限・本人識別子）を検証し、検証できない場合はアクセスを拒否しなければならない。
- **FR-004**: システムは、ユーザーの本人識別をサーバー側で確立した認証情報のみから決定し、リクエストに含まれる任意のユーザー識別子を信用してはならない。
- **FR-005**: システムは、認証プロバイダ上のユーザーと、サービス内のプロフィールを1対1で対応付けなければならない（同一の一意識別子を用いる）。
- **FR-005a**: システムは、同一メールアドレスがGoogleとメール＋パスワードの双方で使われた場合、それらを同一のユーザー（同一の一意識別子・同一プロフィール）にリンクし、データを共有させなければならない。複数の認証方式から同一アカウントへログインできること。
- **FR-006**: システムは、初回ログイン時に当該ユーザーのプロフィール（表示名・アイコン・既定タイムゾーン）を自動的に作成しなければならない。プロフィール作成は冪等であり、重複を作らないこと。
- **FR-007**: システムは、ユーザーが各自のプロフィール情報を閲覧できるようにしなければならない。
- **FR-008**: システムは、ユーザーがログアウトでき、ログアウト時にサービス側と認証プロバイダ側の双方のセッションを破棄しなければならない。
- **FR-009**: システムは、ユーザーが自分自身のプロフィールのみを参照でき、他ユーザーのプロフィールへアクセスできないようにしなければならない。
- **FR-010**: システムは、認証・ログインの失敗時に、内部情報を露出しないユーザー向けの分かりやすいメッセージを表示しなければならない。
- **FR-011**: システムは、認証に関わる秘密情報（アクセストークン・認証用の鍵・サーバー専用鍵・認証Cookie）を、ログ出力やブラウザ（フロントエンド）へ露出してはならない。
- **FR-012**: システムは、PCおよびスマートフォンのブラウザでログイン・ログアウト・プロフィール確認の主要操作を完了できるようにしなければならない（レスポンシブ対応）。
- **FR-013**: システムは、認証済みの操作に対して濫用防止のためのレート制限を適用しなければならない。

### Key Entities *(include if feature involves data)*

- **ユーザー（認証主体）**: 外部認証プロバイダ上で管理される認証アカウント。一意な識別子を持つ。Googleおよびメール＋パスワードという複数の認証方式（identity）を持ちうるが、同一メールアドレスであれば同一の一意識別子へリンクされる。サービスはこの識別子を本人識別の基点とする。
- **プロフィール**: サービス内でのユーザー表現。認証主体と同一の一意識別子を持ち、表示名・アイコン・タイムゾーン（既定はアジア／東京）を保持する。推し・動画・メモなど今後のすべてのユーザー固有データの所有者となる。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新規ユーザーが、トップ到達からログイン完了して自分のホーム画面に入るまでを、通常2分以内・3ステップ以内で完了できる。
- **SC-002**: 認証されていないアクセスが保護画面のコンテンツに到達する割合は0件である（すべてログインへ誘導される）。
- **SC-003**: あるユーザーが別ユーザーのプロフィールを閲覧できる割合は0件である。
- **SC-004**: 初回ログインしたユーザーの100%について、重複のないプロフィールが1件だけ自動作成される。
- **SC-005**: ログアウト後に保護画面へアクセスできるケースは0件である。
- **SC-006**: PC・スマートフォンの双方で、ログイン／ログアウト／プロフィール確認の主要操作が完了できる。
- **SC-007**: 新規ユーザーが、Googleとメール＋パスワードのいずれの方法でもアカウントを作成しログインを完了できる。
- **SC-008**: パスワードを忘れたユーザーが、再設定手続きを通じて新しいパスワードでログインを回復できる。
- **SC-009**: メール確認が未完了のアカウントが保護機能にアクセスできるケースは0件である（確認完了が必須）。
- **SC-010**: 同一メールアドレスをGoogleとメール＋パスワードの双方で利用したユーザーについて、参照されるプロフィールは常に1件（同一）である。

## Assumptions

- MVPのログイン方法は **Googleログイン** と **メールアドレス＋パスワード** の2方式を採用する（ユーザー判断による決定）。これは開発指示書 §5.1（メールはマジックリンク／OTPを想定）からのスコープ変更であり、メールのマジックリンク／OTPは採用しない。指示書 §5.1 は別途この決定に合わせて更新することが望ましい。
- パスワードの保管・ハッシュ化・強度検証の一部は認証プロバイダ（Supabase Auth）の機能に委ねる。サービス側はパスワードを保持・ログ出力しない。
- 認証主体は Supabase Auth、ユーザー識別子はUUIDを用いる（憲法・開発指示書 §5 に準拠）。
- 本フィーチャーは基盤整備（ローカル開発環境、PostgreSQL、CSS基盤、CI、コーディング規約、`.env.example` 等の Phase 0 項目）を伴うが、これらは利用者向け機能ではないため成功基準には含めず、後続フィーチャーが依存する前提条件として整備する。
- 既定タイムゾーンはアジア／東京とする。プロフィールの編集（表示名変更等）は最小限とし、本フィーチャーでは少なくとも閲覧と自動初期化を満たす。
- 推し・チャンネル・動画・メモなどの機能は本フィーチャーの範囲外であり、後続フィーチャー（002以降）で扱う。
- 公開前に必要なプライバシーポリシー・利用規約の整備は、最終フィーチャー（hardening）で扱う。
